O que é o Simulador Esparso?

O simulador esparso utiliza uma representação esparsa de vetores de estado quântico. Um estado quântico esparso é aquele em que a maioria dos coeficientes de amplitude é zero. Em vez de alocar memória para todas as 2ⁿ amplitudes de um sistema de n qubits, o simulador armazena apenas os estados com amplitude não-nula.

Isso permite que o simulador esparso minimize drasticamente o uso de memória, habilitando simulações com um número muito maior de qubits do que um simulador de estado completo (full-state) conseguiria. Simuladores de estado completo desperdiçam memória e tempo em um número exponencialmente grande de amplitudes zero.

Simulador Full-State: memória = O(2ⁿ) → inviável para n > ~50 qubits
Simulador Esparso: memória = O(k) → onde k = número de estados não-zero
Para estados com poucos termos não-nulos: k ≪ 2ⁿ → ganho exponencial

Referência técnica: Jaques e Häner (arXiv:2105.01533) — base do algoritmo do simulador esparso do Azure QDK.

Sparse vs. Full-State: Representação de Memória

⚡ Simulador Esparso

  • Armazena apenas amplitudes ≠ 0
  • Eficiente para estados como Bell, GHZ e circuitos com poucos termos
  • Suporta simulação de 1000+ qubits em estados esparsos
  • Memória proporcional à complexidade do estado, não ao número de qubits
  • Padrão no QDK (VS Code Extension) desde 2024
  • Permite simular programas maiores com o mesmo hardware

📦 Simulador Full-State

  • Armazena todas as 2ⁿ amplitudes, incluindo zeros
  • Preciso para qualquer estado quântico arbitrário
  • Limitado a ~50 qubits em hardware convencional
  • Memória cresce exponencialmente com n qubits
  • Melhor para programas que exploram superposição global
  • Necessário quando o estado não é esparso
Visualização — Estado de Bell |Φ⁺⟩ (Esparso)

O estado de Bell |Φ⁺⟩ = (1/√2)(|000⟩ + |111⟩) tem apenas 2 de 8 amplitudes não-nulas. O simulador esparso armazena somente esses dois coeficientes — os 6 estados cinza não consomem memória.

Estado de Bell de 3 qubits — barras azuis = amplitudes armazenadas · cinza = zeros (ignorados pelo simulador)

Como Executar o Simulador Esparso

O simulador esparso é o simulador local padrão na extensão Microsoft Quantum Development Kit (QDK) para VS Code. Como chamá-lo depende do ambiente:

AmbienteComo chamar o simulador esparso
Arquivo Q# ou OpenQASM no VS CodeExecute o arquivo diretamente (padrão)
Python com biblioteca qdkqsharp.run ou openqasm.run ou qiskit.QSharpBackend
Notebook Jupyter (célula %%qsharp)Chame a operação de entrada: Main()

Exemplo: Estado GHZ de 5 Qubits (sem ruído)

Q# import Std.Diagnostics.*; import Std.Measurement.*; operation Main() : Result[] { let num = 5; return GHzSample(num); } operation GHzSample(n: Int) : Result[] { use qs = Qubit[n]; H(qs[0]); // Superposição no qubit 0 ApplyCNOTChain(qs); // Entrelaça todos os qubits let results = MeasureEachZ(qs); ResetAll(qs); return results; }

Resultado esperado: ~50% das medições em |00000⟩ e ~50% em |11111⟩ — comportamento típico de estado GHZ esparso.

Simulação de Ruído Pauli no Simulador Esparso

O simulador esparso suporta a adição de ruído Pauli para simular os efeitos de decoerência em operações e medições quânticas reais. Use a função ConfigurePauliNoise para definir probabilidades de erro em portas X, Y e Z.

Exemplo: Adicionar 5% de Ruído de Bit-Flip (após alocação)

Q# operation GHzSample(n: Int) : Result[] { use qs = Qubit[n]; // 5% de ruído bit-flip aplicado a TODAS as operações ConfigurePauliNoise(0.05, 0.0, 0.0); H(qs[0]); ApplyCNOTChain(qs); let results = MeasureEachZ(qs); ResetAll(qs); return results; }

Exemplo: Ruído apenas na medição

Q# operation GHzSample(n: Int) : Result[] { use qs = Qubit[n]; H(qs[0]); ApplyCNOTChain(qs); // Ruído aplicado SOMENTE à operação de medição ConfigurePauliNoise(0.05, 0.0, 0.0); let results = MeasureEachZ(qs); ResetAll(qs); return results; }

Exemplo: Ruído segmentado por porta

Q# operation GHzSample(n: Int) : Result[] { use qs = Qubit[n]; // 5% bit-flip apenas na porta Hadamard ConfigurePauliNoise(0.05, 0.0, 0.0); H(qs[0]); // Limpar ruído para o restante do circuito ConfigurePauliNoise(0.0, 0.0, 0.0); ApplyCNOTChain(qs); let results = MeasureEachZ(qs); ResetAll(qs); return results; }

Funções de Ruído Disponíveis — biblioteca Std.Diagnostics
FunçãoDescriçãoExemplo
ConfigurePauliNoise Configura ruído Pauli X, Y e Z com probabilidades independentes. Aplica a todas as portas e medições subsequentes. Substitui configurações anteriores. ConfigurePauliNoise(0.1, 0.0, 0.5)
BitFlipNoise Configura apenas ruído de porta X (bit-flip) com a probabilidade especificada. ConfigurePauliNoise(BitFlipNoise(0.1))
≡ ConfigurePauliNoise(0.1, 0.0, 0.0)
PhaseFlipNoise Configura apenas ruído de porta Z (phase-flip) com a probabilidade especificada. ConfigurePauliNoise(PhaseFlipNoise(0.1))
≡ ConfigurePauliNoise(0.0, 0.0, 0.1)
DepolarizingNoise Configura ruído como porta X, Y ou Z com probabilidades iguais (despolarizante). ConfigurePauliNoise(DepolarizingNoise(0.06))
≡ ConfigurePauliNoise(0.2, 0.2, 0.2)
NoNoise Redefine o modelo de ruído para zero (sem ruído). Útil para limpar configurações anteriores. ConfigurePauliNoise(NoNoise())
≡ ConfigurePauliNoise(0.0, 0.0, 0.0)
ApplyIdleNoise Aplica o ruído configurado manualmente a um único qubit específico durante a simulação. ConfigurePauliNoise(0.1, 0.0, 0.0);

D-Wave — Proof of Quantum Work (PoQW)
Publicado em Physical Review A · Primeiro demo distribuído com 4 QPUs (Canadá + EUA)

A abordagem da D-Wave substitui o hashing clássico (SHA-256) por hashes gerados e validados por computação quântica. Em vez de usar CPU/GPU para resolver puzzles arbitrários de proof-of-work, o sistema mapeia funções matemáticas para spin glasses programáveis simulados nos QPUs de recozimento quântico — criando um hash que só pode ser gerado e verificado eficientemente por hardware quântico.

QUIP Network — Proof of Useful Work (PoUW)
Postquant Labs · Testnet lançado em 2 de abril de 2026 · 13.000+ participantes

A QUIP Network opera em duas camadas independentes: uma camada de consenso-computação onde mineradores resolvem problemas reais usando QPUs, e uma camada de ativos onde carteiras existentes (MetaMask, Ledger, Trezor etc.) são protegidas com assinaturas pós-quânticas. Em vez de puzzles sem utilidade, o PoUW resolve problemas de otimização, simulação e machine learning com valor comercial real.

⛏ Camada 1 — Compute-Consensus
  • ▸Mineradores submetem soluções de otimização real
  • ▸D-Wave Advantage2 compete com CPUs, GPUs e ASICs
  • ▸Reward em tokens QUIP (crédito de computação)
  • ▸~300× menos energia que PoW clássico (Bitcoin)
  • ▸Problemas: roteamento, portfólios, química, ML
🔐 Camada 2 — Asset Layer (PQC)
  • ▸Carteiras existentes ganham assinatura pós-quântica
  • ▸Co-signatário quântico-resistente em cada TX
  • ▸20.000+ carteiras PQC deployadas (EVM + Solana)
  • ▸Proteção "harvest now, decrypt later"
  • ▸Bitcoin, Solana, EVM em roadmap

🌐 Sistemas de Transmissão — D-Wave & QUIP vs. QKD Tradicional

É importante entender que a D-Wave e a QUIP Network utilizam infraestrutura clássica de internet como meio de transmissão — o diferencial quântico está na computação (hashing e otimização via QPU) e na criptografia (PQC), não no canal físico de comunicação. Isso contrasta com sistemas QKD que utilizam fibra óptica quântica ou satélites para distribuição de chaves.

Resumo: D-Wave e QUIP Network protegem via computação quântica (geração de hash) e criptografia pós-quântica (PQC), transmitindo por internet clássica. Sistemas QKD (bancos como HSBC, DBS, PBOC) usam canais quânticos físicos (fibra/satélite) para distribuição de chaves. As duas abordagens são complementares — o ideal a longo prazo é combinar PoQW/PQC (disponível hoje) com QKD (em escala global futura).
